# Devpost Submission Copy

## Project Title

Ouija

## Tagline

A student-safe AI experiment interpreter for middle and high school science labs.

## Inspiration

Students often leave a lab with a table of numbers, a graph, and no clear way to reason from evidence. Generic chatbots can jump straight to writing the lab report, which hurts learning and academic integrity. Ouija was built to help students understand the experiment, check their data, and write their own claim.

## How It Uses AI

Ouija uses a hybrid strategy. Deterministic templates make eight common school-lab demos reliable without credentials. Judge Demo Path condenses each run into five evaluator-visible steps: problem fit, AI design, student workflow, evidence handoff, and submission proof. AI Runtime Proof exposes the live deployment's AI path through `/api/runtime-proof`, including template count, fallback or OpenAI web-search readiness, evaluation coverage, server-only key boundary, and MCP bridge mode without exposing secrets. Model Strategy ranks candidate templates, shows matched signals, reports confidence and candidate gaps, names fallback behavior, and lists risk controls. AI Architecture Map turns that strategy into a readable system diagram: intake, classifier, grounding, data engine, learning guard, evidence bridge, inputs, outputs, and evaluation contracts. Technical Depth Proof turns that into a judge-readable scorecard for decision trace, evaluation harness, grounding quality, expected-pattern engine, privacy, and integrity. AI Evaluation Harness scores classifier confidence, coverage, source grounding, pattern validation, repeat reliability, row validators, safety/integrity, and fallback boundaries for each run.

Data Handling Ledger makes the privacy model inspectable by showing how descriptions, table data, browser-local saved labs, grounding sources, and the server API key are used, retained, and controlled by the student. AIYES Rubric Fit turns those same signals into concrete evidence for the three official criteria: problem/relevance, AI technical design/model strategy, and UX/design. Student Impact Brief makes the problem/relevance criterion first-scan visible by naming the student user, lab-reasoning pain point, before/after benefit, evidence basis, and remaining anonymous-pilot proof gap. AIYES Values Fit maps every run to democracy, diversity, connectivity, innovation, and ethical inclusion using the same concrete product evidence. AIYES Development Journey maps each run to problem identification, data handling, model integration, app build, testing, UX, ethics, impact, constraints, and hosted submission proof. Top Award Radar makes the council audit explicit: Ouija is submittable and competitive, but the app still names external judging, Devpost submission, roster, and pilot-evidence quality risks instead of promising a win. Official AIYES Rules Snapshot keeps that verdict tied to the current Devpost page by showing ages 13-18, students only, the listed 2-5 team requirement with individual/team wording caveat, September 1 deadline, Track 1 required artifacts, judging criteria, award bands, source link, July 18 verified date, and 86-participant snapshot. AIYES Team Readiness Worksheet reduces the roster gap by preparing anonymous role coverage, eligibility, consent, and Devpost-account readiness without storing names or contact details in Ouija. AIYES Judge Q&A Prep keeps live judging answers grounded in the actual app: problem proof, AI architecture, integrity guardrails, UX evidence, pilot-evidence limits, and external steps.

UX and Accessibility Proof makes the User Experience and Design criterion explicit: the default Student view is focused, Judge mode is navigable, responsive/no-overflow behavior is tested across browsers, controls are labeled, citations are clickable, and integrity prompts keep the final claim student-authored.

AIYES Submission Gate makes the submittability claim auditable: deck, video, source/deploy link, source ZIP fallback, Track 1 fit, impact evidence, AI design, and UX are marked as pass/review/external statuses, while Team Readiness Worksheet prepares anonymous roster details and final Devpost submit stays explicitly external.

Pre-Lab Design Coach converts the classification into a before-data checklist for variables, controls, repeats, table columns, source task, safety gate, and a blank hypothesis starter. Pattern Archetype Coach handles off-template labs by choosing a practical graph shape and source question without inventing a final claim. Learning Impact Loop converts the result into measurable student-readiness signals: outcome, data quality, concept learning, integrity, whole-pattern evidence, repeat reliability, and next-trial status. Student Pilot Study Kit turns that impact claim into a practical 10-minute anonymous pilot protocol with a research question, three-session sample plan, run script, success thresholds, consent stop rules, metrics, observer checklist, and evidence to collect. Pilot Evidence Tracker turns the protocol into honest browser-local observations, starts empty with a warning not to claim completed user testing until data exists, and now includes a 100-point quality gate plus a copyable CSV-ready export with quality checks and automatic direct-contact redaction for Devpost, Sheets, Forms, or Notion handoff. Learning Exit Ticket turns that feedback into student reflection prompts judges can inspect: variables, graph pattern, and next step. Student Level Lens adapts the same analysis for two real users inside the stated age range: middle school students get pattern-reading sentence starters, while high school students get quantitative evidence, controls, repeats, and uncertainty prompts. Concept Mastery Check adds scored student understanding proof for the independent variable, expected pattern, and academic-integrity boundary before evidence export. Student Reflection Workspace then captures only student-authored draft answers and marks empty or too-short drafts for revision.

Progress Portfolio turns Saved Labs into repeated learning evidence by showing saved-run count, score trend, subject breadth, strongest run, milestones, and next portfolio action; Portfolio Story Builder turns that evidence into prompts and blanks for a student-written progress story. MCP Integration Coach now uses a server-side dry-run bridge to validate how the student-owned packet, pilot evidence summary, and reflection drafts could become a Source Scout-backed Composio Search source audit, Composio Scholar claim check, Semantic Scholar reference check, Composio Browser source-page capture, DeepWiki public-source proof for `rushtanu14/ouija`, Canvas assignment-context import, Google Docs evidence packet, Google Sheets data log, Google Drive archive, Google Classroom pre-lab checkpoint, Google Forms readiness check, Google Calendar next-trial reminder, or Notion learning record. Source Scout names the actual read-only discovery path (`COMPOSIO_SEARCH_WEB`, `COMPOSIO_SEARCH_FETCH_URL_CONTENT`, `COMPOSIO_SEARCH_SCHOLAR`, `BROWSER_TOOL_CREATE_TASK`, `BROWSER_TOOL_WATCH_TASK`) and explicitly withholds raw rows, reflections, identifiers, and final claims. It now includes July 18 Composio Search receipts for the official AIYES/Devpost source and classroom-safe references for projectile motion, reaction rates, enzyme activity, and water filtration, while the DeepWiki MCP receipt says the public repo must be indexed before live architecture proof is claimed. It also prepares a scoped Composio Sessions ticket that can create a real server-side session only when server credentials, a non-PII session user id, allowed tools, live-export mode, and consent exist. The new session strategy names the safe read-only source verification bundle across Search, Scholar, Semantic Scholar, Browser, and DeepWiki, adds a separate read-only Canvas assignment-context bundle, then separates later Docs/Sheets/Drive/Classroom/Forms/Calendar/Notion exports behind destination review and consent. The public demo stops before external writes and withholds raw MCP URLs from browser responses. The readiness matrix makes required env vars, allowed tools, least-privilege scopes, data shared, consent gates, dry-run checks, and session scope visible.

Grounding Audit scores citation visibility, source agreement, and mixed-evidence boundaries before students use the expected pattern. Expected overlay points are generated from each trusted template pattern, so students visually compare their data against the expected result before reading the coaching text. Pattern Evidence Engine applies deterministic trend, peak, and ratio checks tailored to the classified lab, so a student sees whether the dataset actually supports the expected science pattern before writing a claim. Custom Lab Triage and Pattern Archetype Coach turn low-confidence descriptions into a structured investigation planner plus graph-shape guidance instead of a fake answer. When `OPENAI_API_KEY` is configured, the server can call the OpenAI Responses API `web_search` tool to enrich expected-results explanations and citations. The frontend never receives the API key. A live Reasoning Trail exposes the pipeline: classify the experiment, expose model strategy, run AI evaluation, guide the judge demo, map variables, audit source grounding, audit data handling, overlay expected graph values, guide the student's next action, build a concept scaffold, check the learning exit ticket, check school-lab safety boundaries, import or audit table data, score whole-pattern evidence, check repeated-trial averages and spread, plan the next trial, and coach a student-owned claim. A server-side Deterministic Regression Suite checks supported-template and boundary behavior through `/api/evaluate`; it is not an independent scientific-accuracy or student-outcome evaluation.

## What I Learned

The hardest part was making an AI science helper useful without turning it into an answer machine. The product became stronger when it focused on whole-pattern evidence, method quality, repeated trials, safety, concept understanding, and student-owned claims.
